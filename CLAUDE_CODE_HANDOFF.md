# Claude Code Project Handoff Prompt

Paste the prompt below into Claude Code while its working directory is this repository.

---

You are collaborating on **Nicky's Hiking Pal**, a local-first, mobile-and-desktop hiking journal, planner, map, analytics dashboard, spreadsheet importer, and optional multi-user cloud-sync app.

Repository: `https://github.com/SubchiBeats/nickys-hiking-pal`

Live site: `https://subchibeats.github.io/nickys-hiking-pal/`

## Your role

Act as an independent senior engineer and comparative tester. Inspect the repository before making assumptions. Test existing behavior, identify reproducible bugs or usability problems, and make focused improvements when requested. Preserve working functionality and existing user data.

Do not merely provide a plan when implementation is feasible. Make changes, test them, and clearly report what changed and what remains uncertain.

## Product priorities

1. Extremely user-friendly for nontechnical hikers.
2. Works clearly on mobile and desktop.
3. Existing hiking history must remain accurate and portable.
4. Spreadsheet import is a core workflow because the intended user has years of Excel data.
5. Private by default, with reliable per-user cloud sync when signed in.
6. Map markers, analytics, durations, elevation, goals, exports, and comparisons must always derive from real saved data. Never display static demo analytics as if they are current.
7. Accessibility, readable dark mode, and clear error/empty states matter.

## Architecture

- Static dependency-light app: `index.html`, `styles.css`, and `app.js`.
- `cloud.js` handles optional Supabase authentication, per-user journal sync, and hiking groups.
- `supabase/schema.sql` defines tables, row-level security, version snapshots, groups, shared plans, and photo-storage policies.
- `config.js` contains the deployed public Supabase configuration and is generated by GitHub Actions from repository secrets.
- `sw.js` provides the offline PWA cache. Increment its cache version whenever deployed frontend assets change.
- Leaflet, OpenStreetMap/OpenTopoMap, SheetJS, Lucide, Supabase JS, and Open-Meteo are loaded from CDNs.
- App data is local-first in `localStorage`; signed-in users also sync a private complete journal JSON object to Supabase.
- Main data collections are `hikes`, `plans`, `bucket`, `goals`, `settings`, and `meta`.

## Important current behavior

- Hike elevation accepts exact single-foot values.
- Duration uses hours/minutes/seconds and stores `durationSeconds`.
- Excel/CSV import supports header matching, preview, duplicate skipping, and exact duration/elevation.
- Trail-map locations can be selected from place search, entered as coordinates, or derived from Google Maps URLs containing coordinates.
- Changing a form's visible location clears stale coordinates. Selected search results explicitly replace the marker.
- Automatic OpenStreetMap matches are conservative and labeled unconfirmed until reviewed.
- Map markers and saved-trail rows expose location editing.
- GPX route lines remain separate from the editable representative marker.
- Dashboard overview analytics and their explanatory text must update from the same selected period dataset and show truthful empty states.
- Supabase row-level security isolates each user's private journal. Groups only expose explicitly shared plans.

## Safety and data rules

- Never commit, print, replace, or expose secret/service-role credentials.
- The public Supabase publishable/anon configuration is intended for browser use, but do not alter `config.js` unless explicitly asked.
- Never weaken Supabase row-level security.
- Do not delete or reset user data to fix display bugs.
- Preserve unknown fields when editing existing records so future/imported data is not lost.
- Do not scrape AllTrails or other sources without a supported API and compatible terms.
- Do not silently accept vague geocoding results as authoritative.
- Do not rewrite unrelated files or revert changes you did not make.

Also run:

```powershell

git diff --check
```

For browser tests, serve the repository and load each test page. A passing page changes its document title to `PASS` and lists no failed assertions.

Before considering a change complete:

1. Test the specific reported workflow.
2. Run all three browser regression pages.
3. Check mobile layout and dark-mode clarity for visual changes.
4. Confirm existing local/cloud data behavior is preserved.
5. Increment the service-worker cache version for deployed frontend changes.

## Comparative testing assignment

Start by independently reviewing the current implementation and running the complete test suite. Compare the live site with the local code and specifically investigate:

- Accuracy of all analytics, comparisons, records, goals, and period switching.
- Empty-account behavior: no nonzero comparison may appear beside a zero metric.
- Whether map location changes reliably replace stale coordinates and sync across sessions/devices.
- Form usability on mobile and desktop, especially location selection and spreadsheet import.
- Cloud-sync conflict/data-preservation risks.
- Accessibility, keyboard navigation, dark-mode contrast, and clear button affordances.
- Any regressions not covered by current tests.

Lead with concrete findings ordered by severity and include file/line references. Add regression tests for every bug you fix. Keep changes focused and compatible with the current static architecture unless a larger change is genuinely necessary and explained.

First, inspect `README.md`, `app.js`, `cloud.js`, `index.html`, `styles.css`, `supabase/schema.sql`, the GitHub Pages workflow, and all files in `tests/`. Then run the tests and report your findings before or alongside any fixes.
